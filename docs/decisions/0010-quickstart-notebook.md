# 0010. Ship a quickstart Jupyter notebook

Date: 2026-05-18
Status: Accepted

## Context

PR 1-7 shipped library functionality (provider Protocols, integration
adapters, guardrails + narrative extraction, two paper-aligned metrics
layers, two real perplexity providers). External users so far have only:

- The minimal mock-LLM snippet in `README.md`.
- The per-PR ADRs in `docs/decisions/`.
- The unit + integration test suites under `tests/`.

None of these show the full pipeline running end-to-end on a real model
with real attributions. A reader evaluating the library has to assemble
the moving parts from scratch - friction that hides "what the output
looks like" until after install + setup.

## Decision

Ship a single executable Jupyter notebook at
`notebooks/01_quickstart.ipynb` that runs the full pipeline on a vendored
30-row slice of the OpenML German Credit dataset:

1. Load CSV.
2. One-hot encode (20 raw -> 57 features).
3. Train `RandomForestClassifier` + compute SHAP via `TreeExplainer`.
4. Build `TabularExplanationRequest` via `from_feature_importance`.
5. Generate narrative + extract claims via `Explainer.explain` (two
   `AnthropicProvider` calls).
6. Score the extraction via `score_extraction`.
7. Score narrativity via `score_narrativity` with
   `OpenAICompatibleEchoProvider`.

Outputs are committed so GitHub renders the notebook with a complete
walk-through. No new library code; the notebook only uses public API
from PR 1-7.

The slice is regenerated by a one-shot
`scripts/generate_german_credit_sample.py` (deterministic at seed 42);
the script is committed but never runs at notebook execution time.

## Rationale

- **Why now.** Library functionality is feature-complete for tabular
  flows. Readers need to see the output before they `pip install`.
- **Why German Credit.** Classic credit-risk tabular benchmark; small
  (1000 rows, sampled cheaply); used by canonical SHAP demos so users
  recognize it. The 30-row slice keeps the file in the kilobyte range.
- **Why vendored CSV.** OpenML can rate-limit, be slow, or change row
  order. Vendoring removes a network dependency from the notebook and
  pins exact rows. The regeneration script makes the slice reproducible.
- **Why committed outputs.** GitHub renders `.ipynb` only when outputs
  are present. Cleared outputs leave readers staring at code without
  context. Outputs are documented as illustrative - LLM responses and
  perplexity numbers vary run-to-run.
- **Why `claude-haiku-4-5` + `meta-llama/Meta-Llama-3-8B-Instruct-Lite`.**
  Haiku is the cheapest currently-listed Claude model (already pinned in
  our PR 2 cassette tests). Llama-3-8B-Instruct-Lite is Together's
  cheapest open-weight model that returns Shape-B echo logprobs (matches
  our PR 7 parser path) - see ADR 0009. Total notebook cost
  ~$0.001-0.005 per full run.
- **Why instance row 2.** Manually inspected across all 30 rows: row 2
  has a "Good credit risk" prediction (`P(bad) = 0.16`) with five
  non-trivial SHAP contributions, two positive and three negative,
  magnitude spread ~0.08. The dominant signal (a negative checking
  balance pushing toward "bad") is intuitively plausible - a useful
  property for a tutorial reader's trust - and the mixed signs give the
  narrative real push-pull.

## Consequences

- New `notebook` optional extra in `pyproject.toml`: `jupyter`, `shap`,
  `scikit-learn`, `pandas`. Mirrored in `dev`.
- New top-level directory `notebooks/` with the notebook and a
  `data/` subdirectory for the vendored CSV.
- New top-level directory `scripts/` with the one-shot generator.
  Pattern set for future regeneration scripts.
- README rearranged: existing `## Quickstart (skeleton / mock path)`
  renamed to `## Minimal example (mock LLM, no API keys)`; new
  `## Quickstart: end-to-end notebook` section added above it; stale
  `## What's in v0.0.1` replaced with version-agnostic
  `## Current capabilities` summary of the seven shipped PRs.
- Notebook outputs in source control add ~30 KB to the repo. Acceptable.
- Users without `ANTHROPIC_API_KEY` and `TOGETHER_API_KEY` see committed
  outputs on GitHub but cannot re-execute locally. The README's
  mock-LLM minimal example serves the no-keys case.

## Rejected alternatives

- **OpenML download at notebook runtime.** Rejected - adds network
  dependency, slow cold start, non-deterministic ordering, breaks
  offline reading.
- **Cleared outputs ("clean diff").** Rejected - kills GitHub preview,
  raises the bar to "must execute to evaluate." Outputs are illustrative
  anyway, not canonical.
- **Mock-LLM preview path inside the notebook.** Rejected - adds
  branches and obscures the realistic flow. The README's minimal example
  is the no-keys path.
- **Multiple notebooks (one per modality).** Rejected for PR 8 -
  text / image / graph modalities aren't fully shipped yet (only tabular
  is end-to-end complete). Add per-modality notebooks in later PRs as
  those modalities land.
- **Run the OpenML fetch + sample inside the notebook on first run,
  then cache.** Rejected - caching adds a hidden file in `~/.cache/`
  that the user can't inspect, and the first-run experience is still
  slow + online.
