# 0042. Anchor the extraction sign convention on the predicted-class label

Date: 2026-06-25
Status: Accepted

## Context
Building the hybrid notebook (the end-to-end dogfood of `feature_importance_counterfactual`) surfaced a feature-importance extraction defect. On a prediction whose class label carries negative valence ("Bad credit risk"), the judge LLM inverted the `sign` field uniformly: four of five faithfully-narrated features that raised the predicted class's probability (positive SHAP toward the predicted class, ground-truth sign +1) were extracted as sign -1, dropping `sign_faithfulness` to 0.20 while `value_faithfulness`, `rank_correlation`, and `coverage` were all perfect. A standalone feature-importance run on the same instance reproduced the identical 0.20 with the identical inversions, confirming the defect is pre-existing in the feature-importance path, not introduced by the hybrid work. The `sign_faithfulness` metric is correct: it compares the extracted sign against the numeric sign of the contribution's importance, and the notebook signs SHAP toward the predicted class. The extraction prompt's `sign` instruction was the weak point. It read "+1 increases the predicted class, -1 decreases it", stated only in the static system prompt, with no reference to the concrete predicted-class label. When the label reads "Bad credit risk", the LLM fell back to a valence reading (is this feature good or bad for the applicant) and inverted: a feature that raises P(Bad) is "bad for the applicant", so it emitted -1 where the convention wanted +1.

## Decision
Anchor the `sign` convention on the actual predicted-class label, injected per request into the user prompt. The static system-prompt line now points to a `## Sign convention` section of the user prompt for the exact anchoring. `_build_user_prompt` builds that section using the resolved predicted-class label: it states that `sign` is the probability direction relative to the named label, that +1 means the narrative describes the feature as increasing the probability of that label even when the label is an unfavorable outcome, that -1 means decreasing it, and that 0 means no committed direction, and it states explicitly that this is probability direction and not whether the feature is good or bad for the subject, so a feature that makes the named label more likely is +1 regardless of whether that outcome is desirable. Only the feature-importance extraction prompt changes; `_build_cf_user_prompt`, the `sign_faithfulness` metric, and the counterfactual path are untouched. `_EXTRACTION_PROMPT_VERSION` is bumped from "2" to "3".

## Rationale
- The concrete label is the strongest counter to the valence trap. The drift is triggered by the label's sentiment, so naming that exact label in the sign instruction, at the point where the judge assigns signs, keeps it anchored to probability direction. Static generic wording did not prevent the inversion, and there is direct evidence of that: the old prompt stated the correct convention and the LLM inverted anyway, so a behavioral fix has to put the label in front of the model, not restate the abstract rule.
- The fix is scoped to the prompt, where the defect lives. The metric already computes the ground-truth sign correctly; the extractor was the one emitting the wrong value. Fixing the prompt fixes the emitted data without touching the metric or the counterfactual path.
- The version bump is an honesty signal, not a functional switch. No code path branches on `_EXTRACTION_PROMPT_VERSION`; it is stamped onto extractions and carried through onto grades as metadata. Because v3 changes what the `sign` field means relative to v2, results across the change are not sign-comparable, so distinguishing them by version is the honest thing to do, and it follows the convention that every prior bump tracked a prompt-semantic change rather than a wire-format change (the JSON shape is unchanged).
- The remaining imperfect score is the metric working, not a residual bug. After the fix the same instance scores 0.80: the four risk-raising features now extract +1 and match SHAP, and the one remaining mismatch (job = skilled) is a case where the generated narrative itself claimed the feature reduces risk while its SHAP raises it. The extractor faithfully reported the narrative's claim; the metric correctly flagged a real narrative-versus-attribution divergence. A perfect 1.00 would have required the metric to miss that divergence.

## Consequences
- Feature-importance narratives explaining negative-valence predictions now extract `sign` correctly, so `sign_faithfulness` reflects genuine narrative-versus-attribution agreement rather than a label-sentiment artifact. This affects every prediction whose class label reads as an adverse outcome (default, fraud, disease, churn), a common case.
- `_EXTRACTION_PROMPT_VERSION` is "3". The two tests that pin the extractor's emitted version are updated; the fourteen fixture-side "2" strings elsewhere are arbitrary valid version tags that no code reads, so they are left as is to avoid churn for zero behavior change.
- The hybrid notebook (a later commit) is re-executed against the v3 prompt; its feature-importance half now scores on substance, and its summary notes that a sub-1.00 `sign_faithfulness` reflects a real narrative-versus-attribution divergence rather than an extractor defect.
- This fix was found by dogfooding, the same way the counterfactual-fidelity fixes were. Two findings surfaced from the hybrid notebook: this sign inversion, now fixed, and a second finding that in hybrid mode the feature-importance extractor scans the whole two-section narrative and counts a counterfactual-only value as a hallucination. The second is the intended cost of running both extractors over the whole text (the dual-extraction decision), not a defect, and is left as documented behavior; it is recorded here so both findings have a trail.

## Rejected alternatives
- **Sharpen the static system-prompt wording only, without the per-request label.** Rejected: the inversion happened under a system prompt that already stated the correct convention. Generic wording relies on the LLM generalizing to the negative-valence case, which is exactly what it failed to do. The concrete label is what anchors it.
- **Change the metric to tolerate the inversion.** Rejected: the metric is correct. The ground-truth sign is the numeric sign of the contribution, and the narrative that raised the predicted class's probability should score +1. Loosening the metric would hide real narrative-versus-attribution divergences.
- **Do not bump the prompt version.** Rejected: v3 changes what `sign` means in the emitted data, so v2 and v3 results are not sign-comparable. Even though no code branches on the version, leaving it at "2" would let callers compare incomparable result sets silently.
- **Pick a positive-valence instance for the notebook so the defect does not appear.** Rejected: that hides a real bug affecting a large class of predictions. Fixing it makes the library correct and lets the notebook show honest scores.
